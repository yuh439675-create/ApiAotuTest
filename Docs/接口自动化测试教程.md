## 第一章：理解接口测试的本质

接口测试本质就是 **模拟前端发请求，验证后端返回是否正确**。

一个 HTTP 请求由 4 部分组成：

```
┌──────────────────────────────────────────────────────┐
│  1. URL        请求地址    https://api.com/user       │
│  2. Method     请求方法    GET / POST / PUT / DELETE   │
│  3. Headers    请求头      Content-Type, token 等      │
│  4. Body       请求体      JSON / 表单 / 文件           │
└──────────────────────────────────────────────────────┘
                        ↓ 发送到服务器
                        ↓
┌──────────────────────────────────────────────────────┐
│  服务器返回 Response：                                 │
│  1. Status Code   状态码     200 / 401 / 500          │
│  2. Headers       响应头                               │
│  3. Body          响应体     一般是 JSON                │
└──────────────────────────────────────────────────────┘
```

**你做的事情就是：构造请求 → 发送 → 检查响应是否符合预期。**

### 常见状态码

| 状态码 | 含义 | 什么时候出现 |
|--------|------|-------------|
| 200 | 成功 | 正常请求 |
| 201 | 已创建 | POST 创建资源成功 |
| 400 | 请求错误 | 参数格式不对 |
| 401 | 未认证 | 没带 token 或 token 过期 |
| 403 | 无权限 | token 有效但没权限 |
| 404 | 不存在 | URL 路径错误 |
| 500 | 服务器错误 | 后端代码报错 |

## 第二章：HTTP 方法和使用场景

| 方法 | 用途 | 有请求体吗 | 举例 |
|------|------|-----------|------|
| GET | 查询数据 | 没有，参数在 URL 上 | 查用户列表、搜索 |
| POST | 创建数据 | 有，JSON/表单 | 注册、登录、下单 |
| PUT | 修改数据 | 有 | 修改用户信息 |
| DELETE | 删除数据 | 一般没有 | 删除订单 |

### 对应到框架代码

```python
client = api("yhb")

# GET — 查询，参数用 params，会自动拼到 URL 后面 ?page=1&size=10
resp = client.get("user/list", params={"page": 1, "size": 10})

# POST — 创建，参数用 json，放在请求体里
resp = client.post("user/create", json={"name": "张三", "age": 25})

# PUT — 修改，参数用 json
resp = client.put("user/update", json={"id": 1, "name": "李四"})

# DELETE — 删除，参数用 params
resp = client.delete("user/delete", params={"id": 1})
```

### params 和 json 的区别

```python
# params → 参数拼在 URL 后面
# 实际请求: GET https://api.com/user/list?page=1&size=10
client.get("user/list", params={"page": 1, "size": 10})

# json → 参数放在请求体里，Content-Type 自动设为 application/json
# 实际请求体: {"name": "张三", "age": 25}
client.post("user/create", json={"name": "张三", "age": 25})

# data → 参数放在请求体里，Content-Type 为表单格式
client.post("user/create", data={"name": "张三", "age": 25})

# files → 上传文件
client.post("upload", files={"file": open("test.png", "rb")})
```

## 第三章：接口测试用例设计方法（6 个维度）

拿到一个接口文档后，按以下 6 个维度设计用例：

### 3.1 正向用例（必须有）

所有参数都正确，验证接口正常工作：

```python
def test_login_success(self, api):
    """正确的账号密码 → 应该登录成功"""
    resp = api("yhb").post("sys/login", json={
        "username": "admin01",
        "password": "123456",
        "captcha": "666666",
    })
    assert_response(resp) \
        .status_ok() \
        .json_field_equals("code", 0) \
        .json_field_exists("data.loginCode")
```

### 3.2 参数缺失

少传必填参数，验证接口能正确报错：

```python
def test_login_missing_password(self, http):
    """不传密码 → 应该报错"""
    resp = http.post("sys/login", json={
        "username": "admin01",
        # 故意不传 password
    })
    assert_response(resp) \
        .status_ok() \
        .json_field_equals("code", 500)
```

### 3.3 参数错误

传错误的值：

```python
def test_login_wrong_password(self, http):
    """密码错误 → 应该提示密码错误"""
    resp = http.post("sys/login", json={
        "username": "admin01",
        "password": "wrong_password",
        "captcha": "666666",
    })
    assert_response(resp) \
        .json_field_contains("msg", "密码")
```

### 3.4 边界值

测试极端情况：

```python
def test_username_empty(self, http):
    """用户名为空字符串"""
    resp = http.post("sys/login", json={
        "username": "",
        "password": "123",
    })
    assert_response(resp).json_field_equals("code", 500)

def test_username_too_long(self, http):
    """用户名超长（256 个字符）"""
    resp = http.post("sys/login", json={
        "username": "a" * 256,
        "password": "123",
    })
    assert_response(resp).json_field_equals("code", 500)

def test_pagination_zero(self, api):
    """分页参数传 0"""
    resp = api("yhb").get("user/list", params={"page": 0, "size": 0})
    assert_response(resp).status_ok()

def test_negative_number(self, api):
    """分页参数传负数"""
    resp = api("yhb").get("user/list", params={"page": -1, "size": -10})
    assert_response(resp).status_ok()
```

### 3.5 权限验证

不带 token 或带过期 token：

```python
def test_no_token(self, http):
    """不带 token 访问需要认证的接口 → 应该 401"""
    resp = http.get("user/list")
    assert_response(resp).status_is(401)

def test_expired_token(self):
    """带过期的 token"""
    from Common.common_requests import Requests
    client = Requests()
    resp = client.get("user/list", headers={"token": "expired_fake_token"})
    assert_response(resp).status_is(401)
```

### 3.6 幂等性 / 重复操作

同样的请求发两次：

```python
def test_double_create(self, api):
    """创建同名用户两次 → 第二次应该报重复"""
    data = {"name": "唯一用户", "phone": "13800001111"}
    api("yhb").post("user/create", json=data)
    resp = api("yhb").post("user/create", json=data)
    assert_response(resp).json_field_contains("msg", "已存在")
```

### 用例设计总结表

```
对每个接口，按这张表过一遍：

┌──────────────┬───────────────────────────────────┐
│  维度         │  测什么                            │
├──────────────┼───────────────────────────────────┤
│  正向         │  正确参数 → 成功                   │
│  参数缺失     │  少传必填字段 → 报错                │
│  参数错误     │  传错误值 → 报错                   │
│  边界值       │  空 / 超长 / 0 / 负数 / 特殊字符    │
│  权限         │  无token / 错token / 无权限角色     │
│  重复/幂等    │  同请求发两次                       │
└──────────────┴───────────────────────────────────┘
```

## 第四章：断言（最重要的部分）

**断言 = 判断结果是否符合预期。没有断言的测试毫无意义。**

### 4.1 框架所有可用断言一览

```python
from Common.assertions import assert_response

resp = api("yhb").get("user/list")

# ── 状态码断言 ──
assert_response(resp).status_ok()                          # 状态码 == 200
assert_response(resp).status_is(201)                       # 状态码 == 指定值

# ── JSON 字段断言 ──
assert_response(resp).json_field_equals("code", 0)         # 字段值 == 期望值
assert_response(resp).json_field_exists("data.token")      # 字段存在（不为 None）
assert_response(resp).json_field_contains("msg", "成功")    # 字段值包含子串
assert_response(resp).json_list_not_empty("data.list")     # 列表字段不为空
assert_response(resp).json_field_type("data.total", int)   # 字段类型校验

# ── 性能断言 ──
assert_response(resp).response_time_less_than(3000)        # 响应时间 < 3000ms

# ── 日志记录 ──
assert_response(resp).log_result("查询用户")                # 写入日志 + Allure
```

### 4.2 嵌套路径怎么写

假设响应 JSON 如下：

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "list": [
      {"id": 1, "name": "张三", "tags": ["vip", "new"]},
      {"id": 2, "name": "李四", "tags": ["normal"]}
    ],
    "total": 2
  }
}
```

取值路径用 `.` 分隔，数组用数字索引：

| 路径 | 取到的值 |
|------|---------|
| `"code"` | `0` |
| `"msg"` | `"success"` |
| `"data.total"` | `2` |
| `"data.list"` | `[{...}, {...}]` |
| `"data.list.0"` | `{"id": 1, "name": "张三", ...}` |
| `"data.list.0.name"` | `"张三"` |
| `"data.list.0.tags.0"` | `"vip"` |
| `"data.list.1.id"` | `2` |

```python
assert_response(resp) \
    .json_field_equals("code", 0) \
    .json_field_equals("data.total", 2) \
    .json_field_equals("data.list.0.name", "张三") \
    .json_list_not_empty("data.list")
```

### 4.3 什么场景用什么断言

| 场景 | 用哪个断言 |
|------|-----------|
| 接口是否通了 | `.status_ok()` |
| 业务是否成功 | `.json_field_equals("code", 0)` |
| 返回数据是否存在 | `.json_field_exists("data.xxx")` |
| 列表查询有没有数据 | `.json_list_not_empty("data.list")` |
| 搜索结果包含关键词 | `.json_field_contains("data.list.0.name", "关键词")` |
| 接口性能是否达标 | `.response_time_less_than(3000)` |
| 返回数据类型对不对 | `.json_field_type("data.total", int)` |

### 4.4 链式组合（推荐写法）

多个断言可以用 `\` 换行一直 `.` 下去：

```python
assert_response(resp) \
    .status_ok() \
    .json_field_equals("code", 0) \
    .json_field_exists("data") \
    .json_list_not_empty("data.list") \
    .response_time_less_than(5000) \
    .log_result("查询订单")
```

任何一个断言失败，后面的就不会执行，直接报错。

### 4.5 Python 原生 assert

框架断言满足不了的场景，可以用 Python 原生写法：

```python
def test_list_count(self, api):
    resp = api("yhb").get("user/list")
    data = resp.json()

    assert resp.status_code == 200
    assert len(data["data"]["list"]) >= 1
    assert data["data"]["list"][0]["name"] != ""
    assert isinstance(data["data"]["total"], int)
    assert data["data"]["total"] > 0
```

### 4.6 断言失败的报错信息

断言失败时会显示清晰的错误信息，方便定位：

```
AssertionError: 'code' 期望 0，实际 500
AssertionError: 'data.list' 应为非空列表，实际 []
AssertionError: 期望 200，实际 401
AssertionError: 响应 3502ms 超过阈值 3000ms
```

## 第五章：参数化（一份代码跑 N 条数据）

参数化的意思是：**同一段测试逻辑，用不同的数据跑多次**，每组数据算一条独立用例。

### 5.1 方式一：直接写在代码里

最简单，适合数据少的场景：

```python
import pytest
from Common.assertions import assert_response

@pytest.mark.parametrize("username, password, expect_code", [
    ("admin01", "123456", 0),        # 正确密码 → 成功
    ("admin01", "wrong",  500),      # 错误密码 → 失败
    ("",        "123456", 500),      # 空用户名 → 失败
    ("admin01", "",       500),      # 空密码 → 失败
])
def test_login(self, http, username, password, expect_code):
    resp = http.post("sys/login", json={
        "username": username,
        "password": password,
        "captcha": "666666",
    })
    assert_response(resp).json_field_equals("code", expect_code)
```

运行后自动变成 **4 条独立用例**，Allure 报告里分别展示。

### 5.2 方式二：YAML 文件驱动（推荐）

数据和代码分离，加新用例只改 YAML，不动代码。

**数据文件** `Datas/test_user_api.yaml`：

```yaml
- case_name: "创建用户-正常"
  method: POST
  path: "user/create"
  body:
    name: "测试用户"
    phone: "13800138000"
  expect_status: 200
  expect_code: 0

- case_name: "创建用户-名字为空"
  method: POST
  path: "user/create"
  body:
    name: ""
    phone: "13800138000"
  expect_status: 200
  expect_code: 500

- case_name: "查询用户-正常"
  method: GET
  path: "user/list"
  params:
    page: 1
    size: 10
  expect_status: 200
  expect_code: 0
```

**用例文件** `Test_Case/test_user_api.py`：

```python
import os
import yaml
import pytest
import allure
from Common.assertions import assert_response
from Config.config import Config


def load_cases():
    path = os.path.join(Config.Datas_path, "test_user_api.yaml")
    with open(path, "r", encoding="utf-8") as f:
        return yaml.safe_load(f)


CASES = load_cases()


@allure.epic("用户管理")
@allure.feature("用户接口")
class TestUserApi:

    @pytest.mark.parametrize("case", CASES, ids=[c["case_name"] for c in CASES])
    def test_user_api(self, api, case):
        allure.dynamic.title(case["case_name"])
        client = api("yhb")

        if case["method"] == "GET":
            resp = client.get(case["path"], params=case.get("params"))
        elif case["method"] == "POST":
            resp = client.post(case["path"], json=case.get("body"))
        elif case["method"] == "PUT":
            resp = client.put(case["path"], json=case.get("body"))
        elif case["method"] == "DELETE":
            resp = client.delete(case["path"], params=case.get("params"))

        assert_response(resp) \
            .status_is(case["expect_status"]) \
            .json_field_equals("code", case["expect_code"]) \
            .log_result(case["case_name"])
```

**YAML 里加一条数据 → 自动多一条用例，不用改任何代码。**

### 5.3 方式三：JSON 文件驱动

和 YAML 原理一样，只是格式不同：

**数据文件** `Datas/test_cases.json`：

```json
[
  {
    "case_name": "创建用户-正常",
    "method": "POST",
    "path": "user/create",
    "body": {"name": "张三"},
    "expect_code": 0
  },
  {
    "case_name": "创建用户-空名",
    "method": "POST",
    "path": "user/create",
    "body": {"name": ""},
    "expect_code": 500
  }
]
```

**读取方式**：

```python
import json, os
from Config.config import Config

def load_json_cases():
    path = os.path.join(Config.Datas_path, "test_cases.json")
    with open(path, "r", encoding="utf-8") as f:

CASES = load_json_cases()
```

### 三种方式对比

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 代码内 parametrize | 简单直观 | 数据多了代码臃肿 | 数据少于 5 组 |
| YAML 文件 | 可读性好，维护方便 | 需要读文件 | **推荐，日常使用** |
| JSON 文件 | 通用格式 | 不支持注释 | 已有 JSON 数据时 |

## 第六章：用例的执行顺序和依赖

### 6.1 独立用例（大部分情况应该这样）

每条用例 **独立运行**，不依赖其他用例的执行结果：

```python
class TestUser:
    def test_create_user(self, api):
        # 自己发请求，自己断言
        ...

    def test_query_user(self, api):
        # 自己发请求，自己断言，不依赖上面的 create
        ...
```

### 6.2 有依赖的用例（增删改查流程）

有时候需要先创建再查询再删除，用 **类变量** 在用例之间传递数据：

```python
import allure
from Common.assertions import assert_response


@allure.feature("用户 CRUD 流程")
class TestUserCRUD:
    user_id = None  # 类变量，所有用例共享

    @allure.title("步骤1: 创建用户")
    def test_01_create(self, api):
        resp = api("yhb").post("user/create", json={"name": "流程测试"})
        assert_response(resp).status_ok().json_field_exists("data.id")
        # 把 id 存到类变量，给后面的用例用
        TestUserCRUD.user_id = resp.json()["data"]["id"]

    @allure.title("步骤2: 查询刚创建的用户")
    def test_02_query(self, api):
        resp = api("yhb").get(f"user/{self.user_id}")
        assert_response(resp) \
            .status_ok() \
            .json_field_equals("data.name", "流程测试")

    @allure.title("步骤3: 修改用户")
    def test_03_update(self, api):
        resp = api("yhb").put("user/update", json={
            "id": self.user_id,
            "name": "修改后",
        })
        assert_response(resp).status_ok()

    @allure.title("步骤4: 删除用户")
    def test_04_delete(self, api):
        resp = api("yhb").delete("user/delete", params={"id": self.user_id})
        assert_response(resp).status_ok()
```

> Pytest 默认按方法名字母序执行，用 `01_` `02_` 前缀保证顺序。

### 6.3 单用例内多步场景（业务流程测试）

**场景测试**：在一个 test 里按业务顺序连续调用多个接口，上一步的返回值（token、id 等）作为下一步的入参，用 `allure.step` 拆步，报告里会按步骤展示。

- **适合**：一条完整业务流程（如：登录 → 查用户信息 → 用 user_id 查订单）。
- **写法要点**：
  1. 用 `api("用户")` 拿到带 token 的客户端，或先 `login()` 再手动建 `AuthClient`。
  2. 每一步用 `with allure.step("步骤说明"):` 包起来，便于在 Allure 里看步骤。
  3. 从 `resp.json()` 里取下一步需要的字段，再传给下一个请求。

示例见：**Test_Case/test_scenario_demo.py**。运行方式与单接口用例相同：

```bash
python Runner.py Test_Case/test_scenario_demo.py
python Runner.py -k scenario   # 只跑场景用例
```

## 第七章：Fixture 夹具

Fixture 就是 **在用例执行前/后自动做一些事情**，比如登录拿 token、创建测试数据、测完清理数据。

### 7.1 框架内置的 3 个 Fixture

写在用例方法参数里，Pytest 自动注入：

```python
def test_xxx(self, token, api, http):
    pass
```

| Fixture | 类型 | 用法 | 说明 |
|---------|------|------|------|
| `token` | 函数 | `t = token("yhb")` | 返回 token 字符串 |
| `api` | 函数 | `client = api("yhb")` | 返回带 token 的请求客户端 |
| `http` | 对象 | `resp = http.get(...)` | 不带 token 的裸客户端 |

```python
class TestExample:

    def test_with_auth(self, api):
        """需要登录的接口，用 api"""
        resp = api("yhb").get("user/list")
        assert_response(resp).status_ok()

    def test_without_auth(self, http):
        """不需要登录的接口，用 http"""
        resp = http.get("public/health")
        assert_response(resp).status_ok()

    def test_check_token(self, token):
        """只需要拿 token 字符串"""
        t = token("yhb")
        assert t is not None and len(t) > 10
```

### 7.2 自己写 Fixture

在 `Test_Case/conftest.py` 里添加：

```python
import pytest

@pytest.fixture
def new_user(api):
    """创建一个临时用户，测试完自动删除"""

    # ===== 前置动作（yield 之前）=====
    resp = api("yhb").post("user/create", json={"name": "临时用户"})
    user_id = resp.json()["data"]["id"]

    yield user_id   # ← 把 user_id 传给用例

    # ===== 后置动作（yield 之后）=====
    api("yhb").delete("user/delete", params={"id": user_id})
```

用例里直接用：

```python
def test_update_user(self, api, new_user):
    # new_user 就是 fixture yield 出来的 user_id
    resp = api("yhb").put("user/update", json={
        "id": new_user,
        "name": "改名了",
    })
    assert_response(resp).status_ok()
    # 用例结束后 fixture 自动执行 yield 后面的删除操作
```

### 7.3 Fixture 作用域

```python
@pytest.fixture(scope="function")   # 每条用例执行一次（默认）

@pytest.fixture(scope="session")    # 整个测试会话只执行一次
```

举例：

```python
@pytest.fixture(scope="session")
def admin_token(token):
    """整个测试过程只登录一次，所有用例复用"""

```

    执行测试用例
        ↓

### Step 2 — 写用例文件

```python
import os
import yaml
import pytest
import allure
from Common.assertions import assert_response
from Config.config import Config


def load_cases():
    path = os.path.join(Config.Datas_path, "test_order.yaml")
    with open(path, "r", encoding="utf-8") as f:
        return yaml.safe_load(f)


CASES = load_cases()


@allure.epic("电商业务")
@allure.feature("订单模块")
class TestOrder:
    """数据驱动测试：YAML 里加一条数据 = 多一条用例"""

    @pytest.mark.parametrize("case", CASES, ids=[c["case_name"] for c in CASES])
    @pytest.mark.smoke
    def test_order_api(self, api, case):
        allure.dynamic.title(case["case_name"])
        client = api("yhb")

        if case["method"] == "GET":
            resp = client.get(case["path"], params=case.get("params"))
        elif case["method"] == "POST":
            resp = client.post(case["path"], json=case.get("body"))
        elif case["method"] == "PUT":
            resp = client.put(case["path"], json=case.get("body"))

        assert_response(resp) \
            .status_ok() \
            .json_field_equals("code", case["expect_code"]) \
            .log_result(case["case_name"])


@allure.epic("电商业务")
@allure.feature("订单模块")
class TestOrderFlow:
    """完整流程测试：创建 → 查看 → 取消"""

    order_id = None

    @allure.title("流程: 创建订单")
    def test_01_create(self, api):
        resp = api("yhb").post("order/create", json={
            "product_id": 100,
            "quantity": 2,
        })
        assert_response(resp) \
            .status_ok() \
            .json_field_exists("data.orderId")
        TestOrderFlow.order_id = resp.json()["data"]["orderId"]

    @allure.title("流程: 查看订单详情")
    def test_02_detail(self, api):
        resp = api("yhb").get(f"order/{self.order_id}")
        assert_response(resp) \
            .status_ok() \
            .json_field_equals("data.quantity", 2)

    @allure.title("流程: 取消订单")
    def test_03_cancel(self, api):
        resp = api("yhb").put("order/cancel", json={
            "order_id": self.order_id,
        })
        assert_response(resp) \
            .status_ok() \
            .json_field_equals("code", 0)
```

## 第十章：知识地图总结

```
接口自动化测试知识体系
│
├── 1. HTTP 基础
│   ├── 请求 = URL + Method + Headers + Body
│   ├── 响应 = Status Code + Headers + Body
│   ├── 方法: GET(查) POST(增) PUT(改) DELETE(删)
│   └── 状态码: 200成功 401未认证 403无权限 500服务器错误
│
├── 2. 用例设计（6 个维度）
│   ├── 正向用例       正确参数 → 成功
│   ├── 参数缺失       少传必填字段 → 报错
│   ├── 参数错误       传错误值 → 报错
│   ├── 边界值         空 / 超长 / 0 / 负数 / 特殊字符
│   ├── 权限           无token / 错token / 无权限角色
│   └── 幂等/重复      同一请求发两次
│
├── 3. 断言（8 种）
│   ├── .status_ok()                      状态码 200
│   ├── .status_is(code)                  指定状态码
│   ├── .json_field_equals(path, val)     字段等于
│   ├── .json_field_exists(path)          字段存在
│   ├── .json_field_contains(path, sub)   字段包含子串
│   ├── .json_list_not_empty(path)        列表非空
│   ├── .json_field_type(path, type)      类型校验
│   └── .response_time_less_than(ms)      响应时间
│
├── 4. 参数化（3 种方式）
│   ├── @pytest.mark.parametrize          代码内直接写
│   ├── YAML 文件 + parametrize           数据分离（推荐）
│   └── JSON 文件 + parametrize           数据分离
│
├── 5. Fixture 夹具（3 个内置）
│   ├── token("用户")    → token 字符串
│   ├── api("用户")      → 带认证的客户端（推荐）
│   ├── http             → 裸客户端
│   └── 自定义 fixture   → yield 前=前置, yield 后=后置
│
├── 6. Allure 报告
│   ├── 层级标签: @epic > @feature > @story > @title
│   ├── 严重级别: BLOCKER > CRITICAL > NORMAL > MINOR > TRIVIAL
│   ├── 性能数据: Step 耗时 + 用例汇总表 + 全局统计
│   └── Pytest 标记: @pytest.mark.smoke → -m smoke 过滤运行
│
└── 7. 运行命令
    ├── python Runner.py                   全部用例
    ├── python Runner.py -k keyword        关键字过滤
    ├── python Runner.py -m smoke          按标记过滤
    ├── python Runner.py --parallel        并行执行
    └── python Runner.py -x               首个失败即停
```
